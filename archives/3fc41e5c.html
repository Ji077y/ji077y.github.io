DNS解析与实现探究

本文字数： 5.9k 阅读时长 ≈ 6 分钟

什么是DNS

DNS是互联网核心协议之一，英文全称为：Domain Name System，中文全称为：域名解析服务。

DNS的特点：

  • 是一个将域名和IP地址相互映射的超级分布式数据库。可以让人们更方便的访问互联网
  • 使用TCP和UDP的53端口
  • 使用C/S架构，有客户端和服务器两个部分
  • 服务器一般使用BIND（Berkley Internet Name Domain），客户端使用共享库-由其他应用调用

主机和域名是不同的两个概念

  • www.abc.com 这个是主机名
  • .abc.com 这个是域名

DNS的历史

早期-hosts文件解析方式

在互联网发展的早期，互联网主机数量不多，IP地址难以记住，所以有了对于主机名的需求，希望用“主机名”与服务器通信。这个时候，host文件的文件就暂时解决了这个需求，txt文档里面记录了IP地址与主机名的对应关系。记得小时候去网吧里面玩，如果访问不到某个网站了，网吧的网管就会过来查看下这个host文件是不是没有设置好。

到了后来，互联网主机越来越多，hosts文件无法支撑庞大的数据对应关系时，就出现了DNS服务器。

中期-DNS服务器提供hosts文件

IANA：负责互联网地址分配，管理互联网上面主机名与IP地址的对应关系。

早期的实现： 将解析库（文本文件）通过ftp发布出来。世界各地的Unix主机管理员，设定任务计划，定时的获取解析库文件到本地，实时进行更新。

客户端在有主机名解析请求发生时，通过调用DNS客户端的“共享库”（现在已经成为 glibc 标准库的一部分），去本地的hosts文件获取到解析结果。

后续，从ftp共享发展到了DNS（中心服务器，使用高可用集群避免服务中断），在本地请求发出后，会优先查询本地的hosts文件，然后才会向DNS服务器进行请求。此时，我们需要的是快速的DNS响应请求，默认使用的是UDP协议（不需要进行三次握手，使用53端口）

域名解析结果被缓存

并不需要每次向主机名发起请求，都需要向DNS服务器请求进行名称解析

因为IP地址不会经常变动，在本地DNS客户端第一次得到解析结果后，本地DNS客户端会将解析结果存储在本地，建立缓存。

但是，随着互联网迅猛发展，即使有本地缓存等技术减轻DNS服务器的压力，但是渐渐地一台中心服务器已经难以应付这个工作，这是DNS服务器就进入了下一个阶段：将名称实现分布式应用。

域名解析进入分布式阶段

DNS在这个时期，采用了授权管理机制

  1. 根域
  2. 顶级域
  3. 二级域
  4. 三级域名-建立主机

根域和顶级域由IANA直接管理

二级域，公司，组织和个人可以直接通过域名注册商进行申请注册

  • 二级域申请好了之后，可以直接在域内建立主机
    • 例如申请了077.ink的一级域名
    • 可以自行建立如a.077.ink、b.077.ink等合法主机名

域名结构

网址的域名结构：[protocol]//[hostname][port][path][query][hash]

例如：https://www.077.com:8080/home?id=66#xkgfwei76

对于主机名：www.077.com 本身，是一个树形结构，比较复杂，实际的写法应该是：www.077.com. ，日常我们访问站点的时候，为了方便，不会写最后的.

DNS涉及的角色

角色 英文全称 说明
Client - client 这里单纯的抽象表示一个请求者
DNS 服务器 DNS Server 除了发起请求的 Client，DNS 查询经过的别的节点基本都是 DNS 服务器
DNS 解析器 DNS Resolver 也是DNS服务器的一种
ISP Internet Service Provider 网络服务提供商，例如三大运营商等
根域名服务器 Root Name Server 全球一共13台，负责返回顶级域（.com等）的权威域名服务器地址
顶级域名服务器 Top Level Domain Name Server 负责返回二级域（example.com等）的权威域名服务器地址
权威域名服务器 Authoritative Name Server 负责返回其域名下的Address记录（A record）及其他记录信息

DNS解析器（又称为递归DNS解析器，即DNS Resolver）：会响应客户端的递归请求并话费时间跟踪DNS记录，如上图所示。

  • 它通过发出一系列请求，直到到达请求记录的权威DNS名称服务器为止（或超时或如果找不到记录，则返回错误）来执行此操作
  • 并不是每次都会对所有层级的DNS 域名服务器发送迭代DNS查询请求
    • 第一步通常是会查看自己的缓存里面是否有对应的DNS记录，以此加快DNS解析请求的响应速度

DNS查询方式

递归查询（Recursive query）：A -> B

  • 逐级查询，逐级返回的则为：A -> B -> C -> D

查询说明：

  • 发生在Client发出一个域名解析请求到Server端
  • DNS Server必须返回对应的IP地址，或者返回找不到的报错信息

迭代查询 (Iterative query)：A -> B A -> C A -> D

查询说明：

  • 发生在DNS Server之间，当Client发出域名解析请求之后，DNS Server给予最佳答案
  • 最佳答案可能是距离最近的顶级域名服务器，也可能是权威域名服务器

非递归查询(Non-recursive query)

查询说明：

  • 发生在Client和DNS Server之间，DNS Server已经知道答案，直接返回结果
    1. DNS Server 本机缓存了对应的IP地址，或者缓存了对应域名的权威服务器地址
    2. 只需要再发一次请求，即可拿到结果返回

DNS解析案例

详细过程：

  1. jimmy 在浏览器输入地址：https://077.ink，这个时候，浏览器没有任何缓存，可以理解为使用的是无痕浏览器
  2. 这个时候，浏览器作为C端往Router发送了一个DNS查询递归请求，要求路由器必须返回这个域名对应的IP地址
  3. Router一脸懵逼，啥也不知道，于是向ISP发起对077.ink的递归查询
  4. ISP的DNS Server在收到请求后，会依次向Root Name Server、TLD Name Server、Authoritative Name Server发起迭代查询请求
    1. 根域名服务器（Root Name Server）返回.int的顶级域名服务器的IP地址
    2. 顶级域名服务器（TLD Name Server）返回077.ink的权威域名服务器的Ip地址
    3. 权威域名服务器（Authoritative Name Server）返回最终的077.ink对应的IP地址结果

几点说明：

  1. Client到ISP 之间的DNS查询请求都是递归查询请求
  2. ISP到各个域名服务器之间的请求均为DNS 迭代查询
    • 原因是：递归查询对于服务器的压力要远大于迭代查询
      • 为啥呢？
  3. Router这个角色，一般会使用DHCP协议，ISP会指定一个DNS Server作为Resolver
    • 路由器会拦截这个DNS Server，同时分配自己的IP作为Client端的DNS Server
    • 这个时候，所有的DNS请求都会经过Router
    • 我们也可以自己指定一个DNS，例如google的8.8.8.8，这样请求就不会经过Router

总结，DNS的查询是：递归+迭代，先递归，后迭代进行查询

DNS缓存

DNS缓存的目的是：将数据临时存储在某个位置，从而提高数据请求的性能和可靠性。

一般是存储在靠近客户端的位置，以便更快的响应DNS查询，缩短了加载时间并减少了带宽和CPU消耗。

每个有DNS缓存的服务器，都将存储DNS记录，存储时间由TTL（生存时间确定）

缓存类型

浏览器DNS缓存

当web浏览器缓存了DNS记录时，因为是在本地，所以浏览器缓存是检测请求记录的第一步。

在chrome中，可以在地址栏中输入 chrome://net-internals/#dns查看DNS缓存的状态。

操作系统（OS级）DNS缓存

查找OS的NDS缓存，是Client发送DNS解析请求后在本地进行查找的第二个地方。

OS系统内部用于处理此查询过程通常称为“存根解析器”或者DNS客户端。

步骤：

  1. 当存根解析器从应用程序获取请求时，它首先检查其自己的缓存以查看其是否具有记录
  2. 如果第一步没有记录，则将在本地网络外部发送到ISP内部的DNS递归解析器去查询
  3. ISP的DNS Resolver接收请求后，也会第一时间查找本地缓存

DNS记录类型

DNS服务器创建DNS记录以提供有关域或主机名（尤其是其当前IP地址）的重要信息。最常见的DNS记录类型为：

DNS记录类型 其他 说明
A记录 存储主机名及其对应的IPv4地址 地址映射记录-Address Mapping record
AAAA记录 存储主机名及其对应的IPv6地址 IPv6地址映射记录-IP Version 6 Address record
CNAME记录 用于将一个主机名别名为另一个主机名。当DNS客户端请求包含指向另一个主机名的CNAME的记录时，将使用新的主机名重复DNS解析过程 规范名称记录-Canonical Name record
MX记录 指定域的SMTP电子邮件服务器，用于将传出电子邮件路由到电子邮件服务器 邮件交换器记录-Mail exchanger record
NS记录 指定将DNS区域（例如“ example.com”）委派给特定的权威名称服务器 名称服务器记录-Name Server records
PTR记录 允许DNS解析器提供IP地址并接收主机名（反向DNS查找） 反向查找指针记录-Reverse-lookup Pointer records
CERT记录 存储加密证书-PKIX，SPKI，PGP等 证书记录-Certificate record
SRV记录 服务位置记录，类似于MX，但用于其他通信协议 服务位置记录-Service Location
TXT记录 通常携带机器可读的数据，例如机会加密，发送者策略框架，DKIM，DMARC等，一般用于验证域名拥有者身份 文本记录-Text Record
SOA记录 该记录显示在DNS区域文件的开头，并指示当前DNS区域的权威名称服务器，域管理员的联系详细信息，域序列号以及有关DNS信息频率的信息该区域应刷新 授权开始记录-Start of Authority

设置你的DNS

什么是1.1.1.1

1.1.1.1是一种快速而私密的浏览Internet的方式。它是一个公共DNS解析器，但与大多数DNS解析器不同，1.1.1.1并未将用户数据出售给广告商。1.1.1.1的实现使其成为目前最快的解析器。

为什么要用1.1.1.1

ISP通常有30%-50%的概率会有隐秘的各种劫持动作，不要问我怎么知道的…

运营商https劫持案例

使用第三方DNS解析器，主要原因是：

  1. 安全性
  2. 性能
  3. ISP并不总是在其DNS上使用强加密或支持DNSSEC这将导致ISP的DNS解析器容易出现数据泄露以及让用户面临中间人攻击等威胁
  4. ISP会通过DNS记录来跟踪用户的活动和行为，例如2020年疫情期间，三大运营商的个人足迹查询功能
  5. ISP的DNS解析器通常速度不如第三方DNS解析器，容易过载，在高峰时期，用户体验较差
  6. 某些情况下，攻击者可以通过操作让DNS解析器过载，从而导致DDOS攻击产生
