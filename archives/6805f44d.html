Git分布式版本管理系统教程

发表于 2020-05-20 更新于 2020-05-28 分类于 技术 ， git 阅读次数： Valine：
本文字数： 3.8k 阅读时长 ≈ 4 分钟

0x0 什么是Git&Github

Git是目前世界上最先进的分布式版本控制系统

github是给git服务的，提供了更多的工具，提供多分支开发的服务

SVN与Git的区别

SVN：集中式版本控制系统，版本库放在中央服务器

  • 必须联网才能工作，适合局域网开发，带宽够大，速度够快即可

    • 互联网下，网速慢，就非常蛋疼了！

  • 工作时，必须先从中央服务器拉取最新的版本

  • 工作结束时，把自己做完的活推送到中央服务器去

Git是分布式版本管理系统，每个人的电脑就是一个完整的版本库

  • 无需联网即可干活
  • 例如在自己电脑上修改了文件A，其他人也在自己的电脑上修改了文件A
    • 这个时候，两人之间只需要把各自的修改推送给对方，就可以互相看到对方的修改了！

0x1 Git基础概念

Git工作流程/原理如下图所示：

  • Workspce：工作区
  • Index / Stage：暂存区
  • Repository：仓库区（本地仓库）
  • Remote：远程仓库

0x2 代码开发流程

  1. 系统开发，编写代码
  2. 提交代码到Git本地库，例如自己白天提交了3次到本地库
  3. 晚上一起，将这三次提交到Git远程库，分享给团队其他的人
  4. 第二天起来到公司，为了获取到昨晚加班同事提交的最新代码，要从远程库中拉取一份最新的代码到本地
  5. 在最新的代码基础上继续编写代码
  6. 重复第2步以及以后的步骤直到开发完成

0x3 Git使用前

  • 在文件夹中建立一个Git库
    • git init
      • 该命令会在当前文件夹中初始化一个git库，当前文件夹中的所有文件都可以进行版本管理
  • 设置基础信息
    • git config -l (打印查看配置信息)
    • git config –global user.name “xxx”
    • git config –global user.email “xxx@gmail.com“
      • 需要填写用户名和邮箱作为一个标识
    • git config –global color.ui true (mac版git可用，用于高亮显示)
    • git config -l (再次打印查看配置信息)

–global 参数，这个参数，表示这台机器上所有的Git仓库都会使用这个配置

  • 也可以对某个仓库指定用不同的用户名和邮箱
  • 帮助信息
    • git config –help
    • git help config
    • git help commit
    • git help branch

0x4 开始玩Git

下载与安装

下载安装Git

安装主流使用的操作系统分为Windows和Mac，这里主要是针对Windows进行演示操作，Mac系统的操作是一样的。

安装直接执行默认安装即可，安装完成后，双击windows桌面生成的UI图标即可打开。

这里为了方便，直接使用Cmder也是可以的，以下的操作均用Cmder进行演示。

初始化

新建一个文件夹，命名为jimmygit，用于创建版本库，测试展现，使用命令git –version，查看当前git的版本号：

使用以下命令，进行用户名和邮箱的配置：

git config –global user.name “xxx”

git config –global user.email “xxx”

使用git init 命令把测试目录变成git可以管理的仓库：

git init执行后，会在对应的目录jimmygit中生成一个.git的文件夹，这个千万不要乱动，会破坏git的文件跟踪功能，不想仓库的东西有啥损失的话，就放着不要动就可以了

第一次Git Demo

第0步：新建txt

在jimmygit文件夹中，手动新建一个test.txt文件，里面的内容为：hello,hacker！

这里的txt可以是任何你想跟踪的文件

第1步：使用add添加到Index/Stage

git add test.txt

![](http://pic.077.ink/git add.png?x-oss-process=style/blog_20200322)

没有任何回显则表示add成功了

第2步：使用commit提交到Repository（本地仓库）

git commit -m “test.txt提交”

-m参数，指的是关于你提交的文件的注释说明：

![](http://pic.077.ink/git commit语法查询.png?x-oss-process=style/blog_20200322)

第3步：使用status检查是否有未提交的文件

git status

回显在master分支上，检查结果为：nothing to commit，working tree clean，意思是你的本地工作区文件夹里面的所有文件已经提交了。

第4步：尝试修改test.txt文件再使用status检查一次

这里我将test.txt里面的文件内容修改为：hello,Hackerr!

再次使用status命令进行文件提交检查：

这里回显了Changes not staged for commit，git提示使用add将需要commit提交的修改添加到Stage中去

第5步：查看被修改的内容

git diff <文件名>

文件中把旧的文件和新的文件列出来了，一眼就可以看到改动的地方，Nice！

git跟踪文件版本说明：

  • 只能跟踪文本文件的改动，txt、网页、所有程序的代码等，git会告诉你每次的改动
  • 但是对于图片和视频这类二进制文件，无法进行文件改动跟踪
    • 只能跟踪到图片和视频大小的改变，无法知道修改了什么

第6步：确认修改后，再次add、commit

git add test.txt

git status

git commit -m “文件修改为hello,Hackerr!”

养成良好的习惯，每次commit之前和commit之后，都使用git status进行文件提交情况的检查，以防止有未被提交的修改！

第7步：版本回退

想想一个场景：代码提交后，如果用户反馈有bug，需要紧急回退到上一个版本，这个时候，应该如何操作？我们来具体demo下这个需求的实现吧！

再次会test.txt文件进行修改，修改内容为：hello,0x1023!

git diff

git log

![](http://pic.077.ink/git log查看修改日志.png?x-oss-process=style/blog_20200322)

git log –pretty=oneline

以上的demo已经对文件做了三次提交，如果这个时候，我想回退到第二次提交的文件，要怎么做呢？

第一种回退命令：git reset –hard HEAD^

  • 如果要回退到上上个版本，则使用命令修改为：git reset –hard HEAD^^
  • 如果要回退到第前100个版本的话，则方便命令为：git reset –hard HEAD~100

不知道为啥，居然报错了！

先放着，后面有空再研究是啥原因造成的报错

使用命令 git reset –hard HEAD~<前x个版本号对应的数字>

git reset –hard HEAD~1

0x5 提交出错

failed to push some refs to错误解决方案

回顾整个push过程

一般新手想把自己本地的某个项目关联到github上的仓库（repository）并push代码的流程如下：

  1. 本地项目的目录
  2. 在github上面新建一个repository
  3. 本地关联github上新建的repository
  4. 本地push最新版本的代码到repository

出错点

这个’Initialize this repository with a README’是很多人在新建repository的时候都会勾选的。

  • 勾选了这个选项后，当点击创建仓库的时候，会帮助我们进行一次初始提交
  • 于是在新建仓库里面，默认的就会有README.md和.gitignore（本地同步后看不到，需要打开隐藏文件才能显示）
  • 在以上前提下，我们使用命令’git push -u origin master’就会出现以下报错

为什么会出错呢？

我们在本地执行命令push的时候，在那个时间点，本地和github仓库都是有内容的，但是这两个内容并没有联系。所以，在push（推送）或者pull（拉取）的时候，都会出现没有被跟踪的内容。

当出现未被跟踪内容的时候，在执行拉取或者推送操作，就会出问题！

解决方案

会面临三种情况

  1. 一种是本地存在大量修改过的代码，如果执行解决方案的提议，将github同步到本地，那么会不会出现本来要push的代码全没了的情况？参考评论区
  1. 另一个情况，本地没有代码，那么就不用担心，直接用’git pull –rebase origin master’然后’git push -u origin master’解决。
  1. 新建的时候，不要勾选’Initialize this repository with a README’，参考解决案例
